# Agent instructions for pytboss

`pytboss` is a Python 3.12+ library for controlling PitBoss/Dansons grills
over Bluetooth LE or WebSocket, reverse-engineered from the PitBoss app's
cloud API and the Mongoose OS RPC protocol it runs on. See
[README.md](README.md) for user-facing usage, and its
[Development](README.md#development) section for environment setup and how
to run tests/ruff/mypy locally.

## Gotchas

- **Python version**: the system `python3` may be older than 3.12, but this
  project requires `>=3.12` (see `pyproject.toml`). Use
  `python3.12 -m venv .venv` when creating a virtualenv, not a bare
  `python3 -m venv`.
- **Generated files — edit the generator, not the output**:
  - `pytboss/grills.json` is regenerated nightly by `scripts/dump_grills.py`
    (see `.github/workflows/update-grills.yml`), which pulls grill
    definitions from the PitBoss cloud API. Don't hand-edit the JSON.
  - The "Supported Models" list in `README.md` (between
    `<!-- GRILLS START -->` and `<!-- GRILLS END -->`) is regenerated by
    `scripts/update_readme.py` on release. Don't hand-edit that list.
- **Status/temperature parsing is embedded JavaScript**: `pytboss/grills.py`
  executes JS snippets (ported directly from the grill's own control-board
  firmware) via `dukpy` to parse status/temperature payloads and build
  commands. This is intentional — it preserves parity with the firmware's
  actual logic instead of reimplementing it by hand in Python, so don't
  "clean up" the embedded JS templates without understanding why they exist.
- **`fake_firmware/` and `mos.yml`**: `fake_firmware/` plus the root-level
  `mos.yml` form a fake Mongoose OS firmware image for flashing onto a real
  ESP32, letting the library be tested without a physical grill. `mos.yml`
  must stay at the repo root (not moved into `fake_firmware/`) for the
  Mongoose OS VSCode plugin to find it.

## Before committing

Run the same checks CI runs, and make sure all three pass cleanly:

```sh
pytest
ruff check .
mypy .
```

## Opening PRs

Attach one of the existing GitHub labels (`bug`, `enhancement`,
`documentation`, `breaking`, `dependencies`, `cleanup`, etc.) to each PR —
they drive `release-drafter`'s categorization of the next release's notes
(see `.github/release-drafter.yml`).
